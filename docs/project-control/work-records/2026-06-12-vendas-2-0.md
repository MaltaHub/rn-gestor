# 2026-06-12 — Vendas 2.0: wizard /vendedor/vender + schema v2 + fixes de arquivos

## Contexto

A página placeholder "Simular" foi substituída por **Vender** (`/vendedor/vender`): um
gerenciador de venda completo (wizard) que substitui o dialog básico de venda
(`vehicle-sale-dialog.tsx`, removido). O schema de `vendas` foi reestruturado para
sustentar o fluxo (entradas múltiplas, transferência, cartão, desconto) e alimentar os
templates do Word (`/vendedor/word`). No mesmo ciclo, dois problemas operacionais
reportados foram corrigidos: automação de arquivos não disparava ao vender, e upload de
imagens era sequencial (lento).

## Mudanças de schema (migration `20260612120000_vendas_v2_entradas_pagamento_transferencia.sql`)

- **`venda_entradas`** (nova): entradas múltiplas por venda — `tipo`
  (`pix|cartao_credito|carro_troca|outro`), `valor`, parcelas de cartão,
  `carro_troca_id` (FK carros), `descricao`. RLS on, backend-mediated.
- **`vendas`**: novas colunas `financ_valor`, `cartao_parcelas_qtde`,
  `cartao_parcela_valor`, `desconto`, `tipo_transferencia` (`loja|financiamento|cliente`),
  `valor_transferencia`. Colunas `troca_*` **removidas** (migradas para `venda_entradas`
  com descrição "Troca legada: ..."). `valor_entrada` virou **denormalizada**
  (soma das entradas, trigger `fn_venda_entradas_sync_total`) — locked no grid.
- **`forma_pagamento`** remapeada: `a_vista→a_vista_pix`, `financiado→financiamento`,
  `parcelado→cartao_credito`, `misto→financiamento|a_vista_pix` (heurística por financ_*;
  valor original anotado em `observacao`). Novo CHECK com os 4 codes novos.
- **RPC `fn_vendas_criar_v2(p_venda, p_entradas)`**: cria venda + entradas em transação
  (o trigger de cascade `carros.estado_venda=VENDIDO` participa do rollback).

## Fluxo do carro na troca

Entrada `carro_troca` cadastra um carro REAL: `createCarro` (enrichment por placa + file
automations + audit) e depois `documentos.origem='TROCA'` + `valor_compra=valor da
entrada` (linha de documentos já existe via trigger de criação de carro). Exceção
deliberada de RBAC documentada em `app/api/v1/vendas/route.ts`: VENDEDOR cria carro de
troca pelo fluxo da venda (POST /api/v1/carros continua SECRETARIO+). Se a RPC da venda
falhar, compensação best-effort deleta o carro recém-criado.

## Fix: automação de arquivos no fluxo de venda

`carros.estado_venda='VENDIDO'` é setado por **trigger SQL** ao criar a venda, mas a
movimentação de pastas (fotos → 'Vendidos', documentos → arquivo) é TS
(`ensureVehicleFileAutomations`) e só rodava em `createCarro`/`updateCarro`. Resultado:
**vender nunca movia as pastas** (veículos vendidos presos em 'Fotos dos Veículos').
Fix: `createVenda` e `updateVenda` (quando `estado_venda` muda) agora chamam
`ensureVehicleFileAutomations` best-effort. Estoque legado: rodar
`POST /api/v1/files/automations/reconcile` (ADMINISTRADOR) uma vez.

## Fix: upload de imagens lento

`use-file-upload-flow.ts` processava 1 arquivo por vez E recarregava a árvore de pastas
inteira + pasta ativa **após cada arquivo** (~5x mais requests que uploads). Agora:

- Pool de **4 uploads paralelos** (prepare + PUT ao storage em paralelo).
- **Finalize serializado** via promise chain (dois finalizes concorrentes na mesma pasta
  calculariam o mesmo `sort_order` em `getNextFolderFileSortOrder`).
- Refresh da UI (loadFolders/loadActiveFolder) **uma vez ao esvaziar a fila**.
- Rota `finalize` não monta mais `getFolderDetail` por arquivo (resposta enxuta
  `{folderId, inserted}` — o cliente descartava o detail).
- Retry com backoff re-agenda sem bloquear o pool; abort cancela todos os controllers.

Durabilidade preservada: cada arquivo continua sendo finalizado (persistido) assim que
sobe — sair da página no meio mantém o que já subiu.
